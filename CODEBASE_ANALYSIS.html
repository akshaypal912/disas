🔴 Critical Issues Bugs / Security

Critical
Fake authentication — any password ≥ 6 chars grants coordinator access
src/components/LoginPage.tsx : handleEmailSignIn()

The email/password login route never calls Firebase or any real authentication service. The condition is if (email === 'commander@resp.ai' || password.length >= 6) — meaning any account with a password of 6+ characters is granted full Coordinator (admin) access with a mock token. This bypasses all RBAC entirely.

Any user who clicks "Sign In" with any email and any 6-character password gets a mockToken accepted by the rest of the app as a real session.

SecurityRBAC BypassAuth
Critical
Mock tokens accepted by server API routes — authentication is illusory
server.ts : requireAuth middleware / src/middleware/auth.ts

The Express server uses requireAuth which calls adminAuth.verifyIdToken(token) from Firebase Admin. However, the simulated access path in LoginPage produces tokens like mock_token_coordinator_<timestamp>. These will fail Firebase Admin verification, meaning every API call (/api/logs, /api/feedback, /api/timeline) will return 401 for simulated sessions — the SOS queue sync will silently fail.

SecurityAPIBroken Auth Flow
Critical
IBM watsonx IAM token is a hardcoded string "mock-iam-token" — real LLM calls always fail
backend/app/services/watsonx.py : _get_iam_token()

_get_iam_token() simply return "mock-iam-token" with a comment saying it is a "placeholder mechanism." Even when a real WATSONX_API_KEY is provided, HTTP requests to /ml/v4/generation are sent with Authorization: Bearer mock-iam-token and will be rejected with HTTP 401 by IBM Cloud.

The actual IAM exchange endpoint (https://iam.cloud.ibm.com/identity/token) is mentioned in a comment but never implemented.

AI / LLMIBM watsonxCredentials
Critical
Gemini model name "gemini-3.5-flash" does not exist — all AI triage calls fail
server.ts : lines 286, 431

Both the /api/disaster and /api/severity/classify routes call client.models.generateContent({ model: "gemini-3.5-flash", ... }). The real model identifier is gemini-1.5-flash or gemini-2.0-flash. An invalid model name causes a 404, causing every chat query to silently fall back to the static hardcoded getFallbackDisasterGuidance() function with no indication to the user.

AI / LLMGemini API
Critical
SSL disabled on Postgres connection pool — data transmitted in plaintext
src/db/index.ts : createPool()

The database connection pool is created with ssl: false. Cloud-hosted Postgres instances (Google Cloud SQL, Supabase, AWS RDS) typically require TLS connections. In production this sends SOS events, user credentials, and emergency logs over an unencrypted channel. It will also silently fail to connect on Cloud SQL instances that enforce SSL.

Critical
Health check endpoint hard-codes "connected: true" regardless of real state
backend/app/main.py : health_check()

The GET / health endpoint unconditionally returns "database_connected": True, "llm_connected": True without actually pinging the Supabase database or the watsonx service. Any monitoring system relying on this endpoint will incorrectly report the service as healthy even when both are completely unreachable.

🟠 High Severity Issues Bugs / Broken Features

High
Python backend CRUD endpoints return mock IDs — Supabase is never actually written to
backend/app/api/v1/endpoints/disaster.py, alerts.py, resources.py

All three Python endpoint files return hardcoded mock responses like "id": "mock-disaster-id-abc" and never call get_supabase_client(). The GET list endpoints always return an empty array. The supabase.py client is defined but completely unused in the route handlers.

APISupabaseUnimplemented
High
Two completely separate, incompatible database systems in use simultaneously
src/db/ (Drizzle + Postgres) vs supabase/ (Supabase RLS) vs backend/ (Supabase Python)

The project runs three distinct data stores that are architecturally incompatible:

1. Express server (Node.js) uses Drizzle ORM against a raw Postgres TCP pool with a users/logs/feedback/timeline_events schema.

2. Python FastAPI backend uses the Supabase Python client against a completely different schema (profiles/disasters/resources/alerts/shelters).

3. Two Supabase SQL files define yet another conflicting schema with different table names (users vs profiles, requests vs disasters).

DatabaseArchitecture
High
firebase-admin initialized without service account credentials — token verification will fail
src/lib/firebase-admin.ts

firebase-admin is initialized with only a projectId from the client-side JSON config file. Firebase Admin SDK requires a service account key (JSON) or Application Default Credentials (ADC) for server-side token verification. Without them, every call to adminAuth.verifyIdToken(token) will throw, breaking all protected API routes.
